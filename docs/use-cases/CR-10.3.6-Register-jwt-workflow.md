# CR-10.3.6 — Change Request: Registrierungs-Token von UUID auf JWT umstellen

## Status: **implementiert** ✓

## Kontext

Die bestehenden Use-Cases UC-10.3.1 bis UC-10.3.5 beschreiben den Double-Opt-In-Registrierungsflow.  
Ursprünglich wurde ein `UUID`-Token erzeugt und in einer serverseitigen `Map<UUID, PendingRegistration>` gespeichert.

**Problem:** Die In-Memory-Map wird bei jedem Server-Neustart geleert. Jeder ausstehende Bestätigungslink wird dadurch sofort ungültig (404).

Dieser Change Request ersetzt UUID durch einen **signierten JWT** als Bestätigungstoken und eliminiert damit den serverseitigen Zustand vollständig.

---

## Entscheidung

**UUID → JWT**, stateless.

| Kriterium | UUID + Map | JWT (stateless) |
|---|---|---|
| Serverseitiger Zustand | `Map<UUID, PendingRegistration>` im RAM | keiner |
| Neustart-Sicherheit | Pending-Einträge gehen verloren | Token bleibt gültig |
| Ablaufprüfung | manueller `Instant`-Vergleich | `exp`-Claim, automatisch |
| Manipulationsschutz | UUID ist unratbar | HS256-Signatur, fälschungssicher |
| Duplicate-Prevention | Map-Check beim Register | DB-Check beim Confirm |
| URL-Länge | ~36 Zeichen | ~200+ Zeichen (kein Problem) |

---

## JWT-Payload (Registrierungstoken)

```json
{
  "sub":    "username",
  "email":  "user@example.com",
  "pwHash": "$2a$10$...",
  "type":   "registration",
  "iat":    1718000000,
  "exp":    1718086400
}
```

- `type: "registration"` verhindert, dass ein Auth-JWT als Registrierungstoken missbraucht wird.
- `pwHash` im Token ist sicher — BCrypt ist eine Einwegfunktion; das Klartextpasswort ist nicht rekonstruierbar.
- Gültigkeitsdauer: **24 Stunden** (via `exp`-Claim, keine manuelle Prüfung nötig).

## Umgesetzte Änderungen

### 1. `JWTUtil` — zwei neue Methoden ✓

> Achtung: Das Projekt verwendet **JJWT 0.9.1** (ältere API). Die Methoden nutzen `.setSubject()`,
> `.setSigningKey()`, `.parseClaimsJws()` usw. — nicht die neuere `signWith(Key)`-Variante.

```java
public String generateRegistrationToken(String username, String email, String passwordHash) {
    long now = System.currentTimeMillis();
    return Jwts.builder()
            .setSubject(username)
            .claim("email",  email)
            .claim("pwHash", passwordHash)
            .claim("type",   "registration")
            .setIssuedAt(new Date(now))
            .setExpiration(new Date(now + 24 * 3_600_000L))
            .signWith(SignatureAlgorithm.HS256, SECRET_KEY)
            .compact();
}

public Claims parseRegistrationToken(String token) {
    Claims claims = Jwts.parser()
            .setSigningKey(SECRET_KEY)
            .parseClaimsJws(token)
            .getBody();
    if (!"registration".equals(claims.get("type", String.class))) {
        throw new JwtException("Not a registration token");
    }
    return claims;
}
```

### 2. `PendingRegistration` — auf Record reduziert ✓

**Vorher:**
```java
public class PendingRegistration {
    private final String username;
    private final String email;
    private final String passwordHash;
    private final UUID token;        // entfernt
    private final Instant createdAt; // entfernt
    // Konstruktor + 5 Getter
}
```

**Nachher:**
```java
public record PendingRegistration(String username, String email, String passwordHash) { }
```

> Nebeneffekt: Record-Accessors heissen `username()`, `email()`, `passwordHash()` — **kein `get`-Präfix**.
> `AccountService` musste deshalb ebenfalls angepasst werden (siehe unten).

### 3. `RegistrierungsService` — Map entfernt, JWT-basiert ✓

**Vorher:**
```java
private final Map<UUID, PendingRegistration> pending = new HashMap<>();

public UUID starteRegistrierung(RegisterDto dto) {
    UUID token = UUID.randomUUID();
    pending.put(token, new PendingRegistration(..., token, Instant.now()));
    mailService.sendRegistrationVerification(dto.email(), token);
    return token;
}

public PendingRegistration bestaetigen(UUID token) {
    PendingRegistration reg = Optional.ofNullable(pending.get(token))
            .orElseThrow(TokenNotFoundException::new);
    if (Instant.now().isAfter(reg.getCreatedAt().plus(TOKEN_VALIDITY))) {
        pending.remove(token);
        throw new TokenExpiredException();
    }
    pending.remove(token);
    return reg;
}
```

**Nachher:**
```java
public void starteRegistrierung(RegisterDto dto) {
    if (accountRepository.existsByUsername(dto.username())) {
        throw new UsernameAlreadyExistsException(dto.username());
    }
    String passwordHash = passwordEncoder.encode(dto.password());
    String regToken = jwtUtil.generateRegistrationToken(dto.username(), dto.email(), passwordHash);
    mailService.sendRegistrationVerification(dto.email(), regToken);
}

public PendingRegistration bestaetigen(String token) {
    Claims claims;
    try {
        claims = jwtUtil.parseRegistrationToken(token);
    } catch (ExpiredJwtException e) {
        throw new TokenExpiredException();
    } catch (JwtException e) {
        throw new TokenNotFoundException();
    }

    String username = claims.getSubject();
    if (accountRepository.existsByUsername(username)) {
        throw new UsernameAlreadyExistsException(username); // Link bereits verwendet
    }

    return new PendingRegistration(
            username,
            claims.get("email",  String.class),
            claims.get("pwHash", String.class));
}
```

### 6. `AuthController` — Path-Variable-Typ + Exception-Handling ✓

**Confirm-Endpoint:**

```java
// Vorher: @PathVariable UUID token
@GetMapping("/register/confirm/{token}")
public ResponseEntity<?> confirm(@PathVariable String token) {
    try {
        PendingRegistration pending = registrierungsService.bestaetigen(token);
        Account account = accountService.erstelleAccount(pending);
        TokenData jwt = new TokenData(jwtUtil.generateToken(account.getUsername()));
        return ResponseEntity.status(HttpStatus.CREATED).body(jwt);
    } catch (TokenNotFoundException | UsernameAlreadyExistsException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).build();
    } catch (TokenExpiredException e) {
        return ResponseEntity.status(HttpStatus.GONE).build();
    }
}
```

**Register-Endpoint:** `RegistrationAlreadyPendingException` aus dem catch entfernt (wird nicht mehr geworfen).

## HTTP-Statuscodes nach Migration

| Situation | Code | Grund |
|---|---|---|
| Account erfolgreich angelegt | `201 Created` | JWT im Body, Benutzer direkt eingeloggt |
| Token ungültig / Signatur falsch | `404 Not Found` | `JwtException` → `TokenNotFoundException` |
| Username bereits in DB | `404 Not Found` | Link schon verwendet oder parallele Registrierung |
| Token älter als 24h | `410 Gone` | `ExpiredJwtException` → `TokenExpiredException` |
