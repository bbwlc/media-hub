# UC-10.3.3 — Bestätigungslink via Console (MailService Stub)

## Ziel

Nach einer Registrierung muss der Aktivierungslink an den Benutzer übermittelt werden. In der aktuellen Entwicklungsimplementierung wird der Link **nicht per E-Mail verschickt**, sondern auf der Konsole ausgegeben (Console-Stub).

---

## Implementierung

```java
@Service
public class MailService {
    public void sendRegistrationVerification(String email, String regToken) {
        String link = "http://localhost:8080/auth/register/confirm/" + regToken;
        System.out.println("Registrierungslink fuer " + email + ": " + link);
    }
}
```

### Was passiert hier?

1. Der Aktivierungs-Token (**JWT-String**) wird an die Confirm-URL angehängt.
2. Statt den Link per E-Mail zu senden, wird er mit `System.out.println` in die Konsole geschrieben.
3. Während der Entwicklung kann der Link direkt aus dem Server-Log kopiert und im Browser aufgerufen werden.

> Hinweis: JWT-Token sind deutlich länger als UUIDs (~200+ Zeichen). Die URL funktioniert trotzdem
> problemlos in allen gängigen Browsern und HTTP-Clients.

---

## Einbindung in den Registrierungsflow

```
POST /auth/register
    → RegistrierungsService.starteRegistrierung()  → JWT regToken
    → MailService.sendRegistrationVerification(email, regToken)
    → Konsole: "Registrierungslink fuer user@example.com:
                http://localhost:8080/auth/register/confirm/<jwt>"

GET /auth/register/confirm/{jwt}
    → RegistrierungsService.bestaetigen(jwt)
    → JWTUtil.parseRegistrationToken(jwt)  → Claims (username, email, pwHash)
    → Account in DB anlegen
    → Auth-JWT zurückgeben
```

---

## Warum Console-Stub?

| Grund | Erklärung |
|---|---|
| Kein SMTP-Server nötig | Entwicklung funktioniert ohne externe Mail-Infrastruktur |
| Sofort testbar | Link ist direkt im Terminal sichtbar |
| Token geht nicht verloren | JWT ist stateless — kein Speicher nötig |

---

## Produktionsreife Implementierung

Für den produktiven Einsatz muss `System.out.println` durch einen echten E-Mail-Versand ersetzt werden.

### Option A — Spring Boot Mail Starter (SMTP)

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-mail</artifactId>
</dependency>
```

```java
@Service
public class MailService {
    private final JavaMailSender mailSender;

    public void sendRegistrationVerification(String email, String regToken) {
        String link = "http://localhost:8080/auth/register/confirm/" + regToken;
        SimpleMailMessage message = new SimpleMailMessage();
        message.setTo(email);
        message.setSubject("Registrierung bestätigen");
        message.setText("Bitte klicken Sie auf folgenden Link: " + link);
        mailSender.send(message);
    }
}
```

### Option B — Mailhog (lokaler SMTP-Testserver via Docker)

```bash
docker run -p 1025:1025 -p 8025:8025 mailhog/mailhog
```

Mails werden unter `http://localhost:8025` angezeigt — kein echter Versand, aber realistischer als Console-Output.
